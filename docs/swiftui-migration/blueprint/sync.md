# supabase-sync (`sync`)

**Effort:** XL

## 概述

This subsystem is Fresh Pantry's offline-first sync + family-sharing engine. Local mutations are recorded as immutable SyncOperation records in a Drift-backed FIFO outbox; a single-flight SyncCoordinator drains them to Supabase through SupabaseSyncGateway, which performs full-row or partial-column writes gated by an optimistic-concurrency 'version' column and resolves contended writes with a 3-way, client-patch-wins field merge (merge_policy). RemotePantryRepository owns all Supabase table CRUD, household-management RPCs, and four realtime streams; HouseholdContentSyncCoordinator reconciles local and remote per household (upload local-only rows, subscribe to realtime, merge remote with still-pending local rows) under generation guards. HouseholdSessionController drives email-OTP auth, household create/join/leave/dissolve, and projects the active household into a root-resident provider that every notifier syncs against; background_sync drains the outbox periodically via WorkManager. The wire format (snake_case columns, jsonb payload for recipes/meal-plans, version/client_id/client_updated_at/deleted_at sync columns, soft-delete) and the conflict rules are the parity-critical core.

### lib/sync/merge_policy.dart

_Pure 3-way field-level conflict resolution for a contended remote write._

class MergeResult { Map<String,dynamic> value (deep-frozen); bool conflict; List<String> conflictFields (unmodifiable) }. Function mergeRemotePatch({required Map local, required Map remote, required Map patch, required int? baseVersion, required int remoteVersion}) -> MergeResult. RULES: (1) If baseVersion == null OR baseVersion == remoteVersion => NO conflict, value = {...remote, ...patch} (patch wins, fast path). (2) Otherwise iterate patch.entries: merged starts as copy of remote. For each field: read localValue=local[field], remoteValue=remote[field], patchValue=entry.value. If remoteValue deep-equals localValue OR remoteValue deep-equals patchValue => merged[field]=patchValue, NOT a conflict (no real divergence). Else merged[field]=patchValue AND field added to conflicts list. (3) Return MergeResult(value: merged, conflict: conflicts.isNotEmpty, conflictFields: conflicts). POLICY: LAST-WRITER (client patch) ALWAYS WINS at the field level; conflicts are only *reported* (telemetry), never block the write. Uses recursive deep JSON equality.

### lib/sync/sync_retry_policy.dart

## 迁移注意

"PARITY-CRITICAL INVARIANTS: (1) FIFO + stop-on-first-failure: pushOperations must process ops in created_at order, break on the first error, and return ONLY the ids it acknowledged before the break; the coordinator removes exactly that set. Re-ordering or partial-ack mismatches will drop or double-apply writes. (2) Version is the optimistic lock: every content write sets version=base+1 (or remoteVersion+1 derived from the ACTUAL remote row, never the stale client base) and gates on WHERE version=expected. versionForUpsert NEVER writes 0 (0 means 'local-only, re-upload forever'). First writes (baseVersion<=0) use upsert(ignoreDuplicates:true) so they never downgrade an existing remote version. (3) Conflict policy = client-patch-wins at field level; conflicts are REPORTED (telemetry) but never block — do not introduce blocking/last-modified-wins. The fast path (baseVersion null or == remoteVersion) takes {...remote, ...patch}. (4) Soft delete only: deletes set deleted_at, never physically delete; loads filter deleted_at IS NULL; realtime streams DO NOT filter, so the apply side must drop rows where both 'deletedAt' and 'deleted_at' are non-null (visibleRemoteRows). (5) selectedHouseholdId MUST be a single app-root @Observable; a per-screen/nested instance silently no-ops enqueue (empty household -> no push -> 'other members never see it'). (6) Generation guards everywhere async results land (HouseholdContentSyncCoordinator._isCurrent, refreshHouseholds generation): a household switch or dispose must drop in-flight applies or a stale apply clobbers the current view. (7) _LocalUploadScope: a local row with a pending outbox op targeting a DIFFERENT household must NOT leak into the current household's upload/merge. (8) Id minting: local-only rows keep their non-UUID local id until a household is joined, then get a fresh v4 UUID (syncIdFor / _withXSyncIds / uploadInitialData re-mint). _applyLocalId only writes id when it's a valid UUID so the DB default fills local-only ids. (9) RealtimeSubscribeException must be swallowed (transient) — reporting it as fatal previously caused Sentry crashes FRESH_PANTRY-7/8. (10) clientUpdatedAt/createdAt are UTC ISO-8601. SEQUENCING: HouseholdContentSync._startSync order is load-bearing: upload-local-only -> drain outbox -> subscribe -> bulk pull -> apply, each gated by _isCurrent. DATA-FIDELITY RISK: custom_recipes & meal_plan_entries store the WHOLE domain object in a jsonb 'payload' column (no per-field columns) — the Swift model's full toJson()/fromJson() must round-trip byte-compatibly. GOTCHA: syncClientId is currently the constant 'local-client' for every device, and uploadInitialData uploads inventory/shopping/recipes but NOT meal-plan entries — preserve or consciously fix."}

## 开放问题

- syncClientIdProvider is hardcoded to the constant 'local-client' for ALL devices — client_id is written to every row but is never unique per install, so it cannot currently disambiguate which device made a write. Intentional, or should the Swift port mint a stable per-install UUID (and does any server-side logic / RLS depend on the current constant)?
- The Supabase RPC bodies (list_household_members, accept_household_invite, dissolve_household, etc.) and the RLS policies are server-side and not in this repo — the exact return shapes, permission checks, invite-expiry enforcement, and cascade semantics must be confirmed from the Supabase migrations/SQL before relying on parity.
- uploadInitialData (createHousehold path) uploads inventory/shopping/custom_recipes but NOT meal_plan_entries, whereas the ongoing HouseholdContentSync DOES upload meal-plan entries. Is the initial-adoption omission of meal plans intentional, or a gap to fix in the rewrite?
- intake and deduction operation types are routed through _pushVersionedRow exactly like update for inventory/shopping (and are no-ops for recipes/meal-plan). Is the semantic distinction (intake vs deduction vs update) only meaningful locally, with the server seeing a plain versioned full-row write? Confirm no server-side per-op-type handling is expected.
- Background sync registers a 15-minute WorkManager periodic task, but iOS timing is explicitly described as throttled/unreliable. For the iOS-only Swift rewrite, decide whether BGAppRefreshTask (lighter, ~best-effort) or BGProcessingTask (requiresNetworkConnectivity, longer) is the right primitive, and whether 15min earliestBeginDate is still the target.
- household_invites.expires_at uses 24h for open (email-less) links vs 3 days for email-bound invites — confirm the server enforces expiry on accept (the client only sets it; expiry checking is presumably in the accept RPC).
