# screens-recipes (`screens-recipes`)

**Effort:** XL

## 概述

The recipe subsystem is the app's discovery-to-cooking loop. The Recipes tab (RecipesScreen) presents four ranked lists — Explore (all bundled HowToCook Chinese recipes), Available (recipes scored by current inventory match), Use-Expiring (recipes ranked by how many perishables they clear), and Mine (user's custom recipes) — overlaid with time, category, favorites-only, search, and dietary-exclusion (忌口) filters. RecipeDetailScreen matches each ingredient against inventory (bidirectional substring), supports portion scaling, one-tap add-missing-to-shopping-cart, a tappable cook-step checklist, add-to-meal-plan, and an 'I cooked it' flow that generates inventory-deduction proposals reviewed in DeductionReviewScreen (which logs consumed waste-reduction departures). Custom recipes are authored in a large CustomRecipeFormScreen that also supports AI URL import (paste a recipe link → fetch page → LLM extracts structured JSON → fill form), with RecipeDraftReviewScreen as an alternate AI-draft review surface. All recipe/ingredient matching, ranking, and category logic lives in pure helper functions in recipe_provider.dart, and the explore dataset ships as a bundled JSON asset (assets/recipes/howtocook.json).

### lib/providers/recipe_provider.dart — recipe data + matching helpers

_Loads HowToCook dataset, derives recommended/expiring rankings, and provides all inventory-match / category / dietary helper functions used across recipe screens._

### lib/services/deduction_proposal_factory.dart — DeductionProposalFactory

_Converts a cooked recipe into reviewable inventory-deduction proposals by fuzzy-matching each ingredient to inventory rows._

Static forRecipe(Recipe recipe, List<Ingredient> inventory) → List<DeductionProposal>: per ingredient i, candidates=ProposalPlanner.fuzzyMatchInventoryRows(ri.name, inventory). If empty → DeductionProposal.empty(id='d_<recipeId>_<i>', recipeIngredientName, requiredQty=ri.amount). Else DeductionProposal(id, recipeIngredientName, requiredQty=ri.amount, candidates, chosenIndex=candidates.first.inventoryRowIndex, deductAmount=_initialDeductAmount). _initialDeductAmount(ri, chosen): parse magnitude+unit (structured quantity first, else parseLeadingQuantity(amount)); if magnitude null/<=0 → '1'; if recipe unit and row unit both present and differ → '1' (unit mismatch safety); else formatQuantity(magnitude). Always returns a parseable >0 number.

### lib/services/proposal_planner.dart — ProposalPlanner.fuzzyMatchInventoryRows

_Fuzzy-matches a recipe ingredient name to inventory rows, sorted by soonest expiry, producing DeductionCandidates with stable identity._

fuzzyMatchInventoryRows(name, inventory)→List<DeductionCandidate>: query=name.trim.toLowerCase; matches rows where n==query OR n.contains(query) OR (query.contains(n) && n.length>=2) [length-1 inventory names never match as substring of longer recipe name, e.g. row '蛋' won't match recipe '蛋糕']. Sorts by expiryDate asc (nulls last). Each DeductionCandidate carries inventoryRowIndex (positional, selection key only), inventoryRowId, inventoryRowName, inventoryRowUnit (identity for re-resolve at apply time), displayLabel='<name> <qty><unit>[ · <expiryLabel>]'.

## 迁移注意

Parity-critical invariants: (1) Recipe equality/identity is by id ONLY (used for dedup merges, favorites, hero tags) — preserve. (2) Ingredient↔inventory matching is BIDIRECTIONAL case-insensitive substring, with the guard that a length-1 inventory name (e.g. '蛋') never matches as a substring of a longer recipe name ('蛋糕') in the deduction fuzzy matcher, while the loose direction stays in recipeIngredientMatchesInventory — these two matchers differ and both must be ported exactly. (3) dedupeRecipeIngredients keys on name.trim().toLowerCase() and MUST run at every recipe entry point (JSON load, custom add/update, AI draft) so the in-memory list matches a reload. (4) Portion scaling is display/cart-only — it never mutates stored recipe; scaledBy is a no-op for factor==1 and for non-numeric quantities (适量/ranges). (5) formatQuantity strips float artifacts (2-decimal round, int when whole) — reuse everywhere quantities are rendered/stored. (6) Deduction default amount falls back to '1' on unit mismatch or non-numeric magnitude, and applyDeductionProposals re-resolves rows by stable identity (inventoryRowId preferred, name+unit fallback), aggregates duplicates, never coerces non-numeric stock to 0, and logs FoodLogOutcome.consumed for emptied rows (feeds waste stats) — getting this wrong corrupts inventory or waste metrics. (7) The category filter is auto-cleared if the selected category isn't present in the active tab (treated as 全部) so a stale filter never silently empties a tab. (8) recipesProvider must never throw — fetch failure surfaces as fetchFailed flag, and the explore error/retry UI keys off that. (9) The form's canPop:false discard guard must use an explicit pop (not maybePop) per the documented PopScope re-entry bug. (10) cookingMinutes defaults to 30 and difficulty clamps to 1-5 on AI parse and JSON load. (11) AI baseUrl must end in /v1 (404 hint) and responseFormat json_object is sent for recipe parsing. Sequencing: AI parse → applyRecipeDraft fills form → user edits → save; or RecipeDraftReviewScreen confirms directly. Data-fidelity risk: legacy ingredients stored as amount-only must round-trip through quantity/unit split (_IngredientControllers.from / RecipeIngredient.fromJson) without losing the original amount.
