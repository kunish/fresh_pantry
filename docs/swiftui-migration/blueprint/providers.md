# state-providers (`providers`)

**Effort:** XL

## 概述

The Riverpod layer is the app's view-model + orchestration tier sitting between the Drift/SharedPreferences storage repos and the SwiftUI-equivalent screens. It splits into (1) stateful domain Notifiers that own a list/set/object, persist every mutation through a serialized PersistenceQueue, and enqueue household-sync ops via the SyncEnqueue mixin (inventory, shopping, custom recipes, meal plan, food log); (2) ephemeral "review" Notifiers (intake/deduction proposals) that stage edits then apply-and-clear into the inventory; (3) settings/preference Notifiers backed by SharedPreferences (AI settings, reminder settings, favorites, dietary exclusions); (4) many derived `Provider`/`Provider.autoDispose`/`FutureProvider` selectors that compute filtered/grouped/scored views with zero state; and (5) DI seam providers that throw-by-default and are overridden in main() (database, SharedPreferences, notification service, share/invite sources, seeds). The defining invariant across all DB-backed notifiers: `build()` returns a ONE-SHOT startup seed that is consumed on read, so pull-to-refresh and household-switch MUST call the notifier's own `reload()` (re-reads local DB scoped to active household) and MUST NEVER call `ref.invalidate(...)` — invalidate rebuilds against the already-consumed seed and silently empties the list. Local DB is the durable source of truth; in-memory state is an optimistic mirror with rollback.

## 组件(28)

### providers/_persistence_queue.dart — PersistenceQueue mixin

_Serializes all persistence writes for one notifier so a later write never races an earlier one; isolates write failures._

`mixin PersistenceQueue`. Private field `Future<void> _pendingPersistence = Future.value()`. Single protected method: `Future<void> queuePersistence(Future<void> Function() write, {bool rethrowError = false})`. Implementation: `final result = _pendingPersistence.then((_) => write()); _pendingPersistence = result.catchError((e, st) { debugPrint('PersistenceQueue write failed: $e'); }); return rethrowError ? result : _pendingPersistence;`. Semantics: the internal chain ALWAYS swallows exceptions (only debugPrint) so one failed write cannot stall the queue; when rethrowError=true the RETURNED future still carries the real result/exception so the caller can react (e.g. roll back optimistic state, keep a draft for retry) — but the internal queue keeps running regardless. Swift mapping: a serial actor or an `AsyncChannel`/task-chain that awaits the previous op; surface a throwing variant and a swallowing variant.

### providers/review_notifier_base.dart — ReviewNotifierBase mixin

_Shared apply-then-clear semantics for the staging/review notifiers._

`mixin ReviewNotifierBase<TState> on Notifier<TState>`. Declares abstract `void clear();`. Provides `Future<T> applyAndClear<T>(Future<T> Function() apply) async { final result = await apply(); clear(); return result; }`. Used by IntakeReviewNotifier and DeductionReviewNotifier — apply proposals to inventory, then reset the review state to empty. Note: clear() is called only on success (after apply completes); if apply throws, state is NOT cleared (draft preserved for retry).

## 迁移注意

PARITY-CRITICAL INVARIANTS: (1) One-shot seed + reload(): build()/init must NOT be re-run to refresh; every DB-backed store needs its own reload() that re-reads scoped storage. In Riverpod, ref.invalidate(inventory/shopping/customRecipe/mealPlan/foodLog) on a notifier silently empties the list ('下拉刷新清空' bug). The ONLY legitimate invalidate is backup import's ref.invalidate(addHistoryProvider) — a DERIVED provider, safe. (2) activeHouseholdId MUST be root-resident: a nested override makes enqueueSync read empty household, no-op the enqueue, and silently drop sync writes ('对方看不到'). (3) Optimistic-state + rollback: every mutation sets state first, then queuePersistence(rethrowError:true); on failure restore prior state so memory never diverges from disk. (4) Identity-not-index resolution at apply time: intake merge and deduction resolve the LIVE inventory row by domain identity (name×unit×storage for merge; inventoryRowId-then-name-guarded-index for deduction), never a stale positional index captured at proposal time — the list can reorder/shrink/be restored from a persisted draft across launches. (5) Sync ids: inventory items always carry a stable UUID (blank/non-UUID ids minted fresh) to avoid duplicate rows across household transitions; shopping/custom-recipe/meal-plan use syncIdFor (UUID only when joined to a household). (6) ADR-0001 merge rule: non-perishable name×unit×storage matches merge+sum; perishables always create a new batch (and the review UI must refuse to toggle a perishable into mergeInto). Consolidation runs on load AND on replaceFromRemote, emitting sync update(survivor)+delete(dropped) deltas. (7) quantity is a String (parse + formatQuantity, 2-decimal float-fix) — non-numeric quantities ('适量','半盒') must NEVER be coerced to 0 and deleted during deduction. (8) Delete patches are {'deletedAt': iso8601 UTC}; updates carry baseVersion=remoteVersion for optimistic concurrency. (9) Food log records departures (consumed/wasted, wasExpiring snapshot) on inventory remove/removeMany (optional outcome) and on cooking deduction (auto consumed); undo reverses via logId. wasteRate/rescued semantics matter for the stats screen. (10) recipesProvider must stay non-throwing (failure -> empty list + recipesFetchProvider.fetchFailed). (11) selectedCategory/selectedStorage/sortMode/wasteStatsWindow/shoppingFilter persist across tab switches (they are StateProviders, not autoDispose). (12) NotificationSync single-flight + ref.mounted guards after awaits (UnmountedRefException). SEQUENCING: build the SwiftData container + active-household session first, then stores, then sync outbox/coordinator, then notification sync (depends on inventory+settings being live). DATA-FIDELITY RISK: food log is NOT in backup export/import and is NOT yet household-synced — preserve that boundary or document the change."
